Custom runners execute image builds on infrastructure you control. Starliner schedules build jobs to registered runners instead of using the platform's default build environment.

The [Starliner runner](https://github.com/starlinerapp/runner) is the reference implementation. It clones a repository, builds a Dockerfile inside a Firecracker microVM, and pushes the resulting image to your container registry.

For installation and registration steps, see the [user manual](/user-manual/custom-runners/self-hosted-runners).

## Components

| Component | Role |
| --------- | ---- |
| **API** (`server-api`) | Dashboard, deployments, runner registration (HTTP), build log reads |
| **Builder** (`server-builder`) | All runner gRPC — heartbeats, job claims, log/result reporting, build dispatch |
| **Runner** (`runner` binary) | Runs on your build host; clones repos, builds in Firecracker, pushes images |
| **NATS JetStream** | Async messaging between API and builder (build triggers, job queues, results) |
| **Redis** | Shared runtime state — runner leases, capacity, build logs, dispatch locks |

The runner only talks to Starliner over **gRPC to the builder**. It never connects to NATS or Redis directly. Registration is the one exception: a single HTTP call to the API during `runner register`.

## End-to-end flow

When a deployment triggers an image build:

<Mermaid
  value={`sequenceDiagram
    participant Runner
    participant Builder
    participant NATS
    participant Redis
    participant API

    API->>NATS: build.triggered.{buildId}
    NATS->>Builder: one replica (queue group)
    Builder->>Redis: read capacity, acquire dispatch lock
    Builder->>NATS: runner.job.{runnerId}

    loop Every ~5s
      Runner->>Builder: StreamHeartbeats (gRPC)
      Builder->>Redis: refresh lease + capacity
    end

    loop Every 2s
      Runner->>Builder: ClaimJob (gRPC)
      Builder->>NATS: pull runner.job.{runnerId}
      Builder-->>Runner: job or empty
    end

    Runner->>Builder: ReportBuildLog (gRPC)
    Builder->>Redis: append build:{id}:logs

    Runner->>Builder: ReportBuildResult (gRPC)
    Builder->>NATS: runner.job.result.{buildId}
    NATS->>Builder: one replica (queue group)
    Builder->>NATS: build.completed.{buildId}
    NATS->>API: one replica (queue group)`}
/>

1. **Trigger** — The API publishes `build.triggered.{buildId}` to NATS.
2. **Dispatch** — One builder replica (see [NATS queue groups](#nats-queue-groups) below) selects an eligible runner, acquires a dispatch lock in Redis, and enqueues the job on `runner.job.{runnerId}`.
3. **Claim** — The runner polls `ClaimJob` over gRPC. The builder pulls the job from NATS and returns it.
4. **Build** — The runner clones the repo, builds inside a Firecracker microVM, and pushes the image (see [Build execution](#build-execution)).
5. **Report** — Log chunks and the final result go back over gRPC. The builder writes logs to Redis and publishes the result to NATS.
6. **Complete** — A builder replica finalizes the result and publishes `build.completed.{buildId}`. The API updates the deployment.

### Runner selection

Before enqueueing a job, the builder picks the least-loaded eligible runner:

1. **Organization runners first** — runners registered to the project's organization
2. **Global runners as fallback** — cluster-wide runners with no organization scope ([created via CLI](/user-manual/custom-runners/global-runners))

A runner is eligible only if it is online (valid heartbeat lease in Redis), not deleted, and below its `max_concurrent_jobs` limit. Among eligible runners, Starliner picks the lowest load ratio (`active_jobs / max_concurrent_jobs`).

The job payload includes everything needed to build and push the image:

| Field | Description |
| ----- | ----------- |
| `git_url`, `branch_name`, `access_token` | Source repository to clone |
| `root_directory`, `dockerfile_path` | Build context within the repo |
| `image_name`, `image_registry_url`, `registry_push_token` | Target registry and push credentials |
| `args` | Docker build arguments |

### Build execution

On the runner host, each job follows this path:

<Mermaid
  value={`sequenceDiagram
    participant Builder
    participant Runner
    participant VM as Firecracker microVM
    participant Registry

    Builder-->>Runner: build job (via ClaimJob)
    Runner->>Runner: Clone repository
    Runner->>VM: Create microVM with BuildKit
    VM->>VM: Build Dockerfile
    Runner->>Registry: Push image with commit SHA tag
    Runner->>Builder: ReportBuildLog + ReportBuildResult`}
/>

Builds run with `sudo` because Firecracker requires elevated privileges to create and manage microVMs.

## Runner lifecycle

Connecting a runner takes three steps: install, register, and start the daemon.

<Mermaid
  value={`sequenceDiagram
    participant Admin
    participant API
    participant Runner
    participant Builder

    Admin->>API: Generate registration token (dashboard or CLI)
    Admin->>Runner: runner install --baseUrl
    Admin->>Runner: runner register --token
    Runner->>API: POST /api/runners/register
    API-->>Runner: 201 Created
    Note over Runner: registration.json saved
    Admin->>Runner: systemctl start starliner-runner
    Runner->>Builder: gRPC heartbeats + job polling`}
/>

### 1. Install

```bash
sudo ./runner install --baseUrl https://starliner.app
```

Writes the control plane URL to `/usr/local/share/runner/config.json` and installs the binary, VM assets, Firecracker, and BuildKit. All later HTTP and gRPC calls use this URL.

### 2. Register

```bash
sudo runner register --token <registration-token>
```

Exchanges a one-time token for a long-lived registration credential via `POST /api/runners/register`. The token expires after one hour and can only be used once. On success, the runner saves `registration.json` locally (mode `0600`). The same token is sent in the `authorization` metadata header on every subsequent gRPC call.

`runner register` prompts for `name` and `max-concurrent-jobs` if not passed as flags. Defaults are the `self-hosted` label and a concurrency limit of `1`.

### 3. Start

```bash
sudo systemctl enable --now starliner-runner
```

`runner start` runs two concurrent loops against the builder over gRPC:

| Loop | gRPC method | Interval | Purpose |
| ---- | ----------- | -------- | ------- |
| Heartbeat | `StreamHeartbeats` | ~5s | Report liveness and capacity; builder refreshes a 10s Redis lease |
| Job polling | `ClaimJob` | 2s | Pick up build jobs when `active_jobs < max_concurrent_jobs` |

When a job is claimed, the runner executes it in the background and reports progress via `ReportBuildLog` and `ReportBuildResult`.

On disconnect, the runner reconnects with exponential backoff from 1 second up to 30 seconds. If heartbeats stop, a builder scheduler marks the runner offline and it will no longer receive new jobs.

## Local files

| Directory | Path | Purpose |
| --------- | ---- | ------- |
| Assets | `/usr/local/share/runner` | VM images and `config.json` (written during `install`) |
| State | `$HOME/.cache/runner` | Registration data, VM registry, per-VM workspaces |

The systemd service sets `HOME=/root`, so the state directory is `/root/.cache/runner`.

| File | Path | Contents |
| ---- | ---- | -------- |
| `config.json` | `/usr/local/share/runner/config.json` | `base_url` |
| `registration.json` | `$HOME/.cache/runner/registration.json` | Token, name, `maxConcurrentJobs` |
| `vms.json` | `$HOME/.cache/runner/vms.json` | VM registry metadata |
| VM workspace | `$HOME/.cache/runner/vms/<id>/` | Firecracker config, sockets, logs |

## Network

The runner host needs outbound HTTPS to your Starliner instance (port 443, or 80 for `http` URLs). gRPC uses HTTP/2 on the same host. No inbound ports are required — the runner initiates all connections.

For self-hosted instances with self-signed TLS, pass `--insecure-skip-tls-verify` to `runner register` and `runner start` (or set it in the systemd unit's `ExecStart`).

## Developing the runner

### Requirements

| Requirement | Notes |
| ----------- | ----- |
| Linux (amd64) | Required for Firecracker microVMs |
| `sudo` | Required for install and builds |
| KVM | Host must support hardware virtualization |

macOS and Windows support is planned but not yet available.

### Build from source

```bash
git clone https://github.com/starlinerapp/runner.git
cd runner
./scripts/build-bundle.sh
cd dist/runner-dev-linux-amd64
sudo ./runner install
```

Use `--cli-only` to rebuild the CLI without rebuilding the guest image:

```bash
./scripts/build-bundle.sh --cli-only
```

After `install`, the binary is at `/usr/local/bin/runner`.

### CLI reference

**`runner install`** — Install CLI, VM assets, Firecracker, and BuildKit. Run once per machine.

**`runner register`** — Register with a one-time token from the dashboard or CLI.

**`runner build`** — Standalone build without Starliner, useful for local debugging:

```bash
sudo runner build \
  --repository https://github.com/org/app.git \
  --branch-name main \
  --github-token "$GITHUB_TOKEN" \
  --image registry.example.com/app \
  --registry-username "$REGISTRY_USER" \
  --registry-password "$REGISTRY_PASSWORD"
```

Images are tagged with the cloned commit SHA.

**`runner vm`** — Manage Firecracker microVMs directly:

| Command | Description |
| ------- | ----------- |
| `runner vm create` | Create a microVM |
| `runner vm list` | List running microVMs |
| `runner vm delete <id>` | Remove a microVM |
